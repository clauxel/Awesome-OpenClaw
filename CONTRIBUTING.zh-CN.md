# 参与贡献 Awesome OpenClaw

**语言版本：** [English](CONTRIBUTING.md) | **简体中文**

感谢你帮助这个列表保持有用。

## 编辑标准

这个仓库应当是 **developer-first，而不是 marketing-first**。

优先收录下面这类资源：

- 官方、技术性强、且不容易很快过时
- 在安装、配置、调试、安全加固或扩展阶段真正有帮助
- 让第一次评估 OpenClaw 的普通开发者也能看懂

尽量避免下面这类资源：

- 以炒作、增长故事或泛新闻报道为主
- 很快会过时、却又缺少明确验证的价格对比表
- 无法核实的规模、使用量、性能或流行度宣称
- 充满联盟营销、但实际上并没有真正教会读者任何东西的托管页面

## 什么内容适合放进 README

- 官方 OpenClaw 文档
- 官方 OpenClaw 仓库
- 高信号的一键部署模板
- 安全指南和加固参考
- 具有明确开发者价值的 skills、plugins、tools 和 integrations
- 能把架构、部署或故障排查讲清楚的深度技术文章

## 多语言维护规则

- 把 `README.md` 和 `CONTRIBUTING.md` 视为结构与意图的英文源文件。
- 只要你修改了面向读者的说明，就要把同样的改动同步到本地化 `README.*.md` 文件和对应的 `CONTRIBUTING.*.md` 文件中。
- 除非维护者明确要求某种语言做特例，否则不同语言之间的章节顺序、徽章、表格、代码块和相对链接都应保持一致。
- 如果你一时只能先更新一种语言，请在 commit 或 PR 中明确注明翻译缺口，以便尽快补齐。

## 如何新增或更新资源

1. 编辑 `README.md`。
2. 将相同的结构性改动同步到每个本地化 `README.*.md` 文件。
3. 把链接放到最小、最贴切的章节里。
4. 添加一条简短说明，聚焦“为什么开发者应该点进去看”。
5. 优先保留少量高价值链接，而不是堆出一大份噪声列表。

### 格式

**用于项目符号列表：**

```markdown
- [资源名称](https://link.example) - 这项资源为什么对开发者有帮助。
```

**用于表格行：**

```markdown
| [资源名称](https://link.example) | 为什么值得看 |
```

## 写作规则

- 官方文档应排在第三方资源前面。
- 多用面向行动的描述，少用空泛形容词。
- 如果某个说法具有时效性，请补充来源或日期。
- 除非经过认真核实并且确实有价值，否则不要重新引入庞大的托管、价格或媒体列表。
- 保持 README 从上到下都易于扫描。
- 保持所有本地化文件里的语言切换行一致。
- `README.md` 中被维护者锁定的表格视为锁定，除非维护者明确要求修改。

## 校验清单

在你添加一个资源之前，先问自己：

- 它是否真的能帮助普通开发者安装、运行、保护或扩展 OpenClaw？
- 链接现在是否还能打开？
- 描述是否足够具体，让读者知道它为什么重要？
- 几个月后它是否仍然值得看？
- 你是否已经把等价改动同步到本地化文件？
- 每个 README 中指向同语言 `CONTRIBUTING` 的链接是否仍然可用？

如果这些问题大多数都是否定答案，那它大概率不适合放在这里。

## 范围

这个目录当前维护的是 Markdown 文档，尤其是 `README.md`、本地化 `README.*.md`、`CONTRIBUTING.md` 以及本地化 `CONTRIBUTING.*.md`。

除非这些文件后来真的被加入仓库，否则不要默认这里存在 `directory.html` 或 `index.html` 之类仅供网站使用的文件。

## 问题反馈

当你发现下面这些问题时，请开 issue 或直接发更新：

- 失效链接
- 过时的安装步骤
- 错误的安全建议
- 会误导人的描述
- 膨胀但不够可执行的章节
